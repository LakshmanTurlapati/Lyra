# Pitfalls Research

**Domain:** Synthetic dataset curation and small LLM fine-tuning (1.7B parameter distillation)
**Researched:** 2026-04-20
**Confidence:** HIGH (multiple peer-reviewed sources, official documentation, community evidence)

## Critical Pitfalls

### Pitfall 1: Single-Source Synthetic Data Homogeneity

**What goes wrong:**
All 5K training samples generated by a single teacher model (Claude Opus) converge on a narrow stylistic and structural distribution. The student model learns to mimic Opus's phrasing patterns, hedging language, and formatting preferences rather than acquiring the underlying capabilities. This is "style imitation" not "capability transfer." Research shows fine-tuning with synthetic data from a single source model produces the weakest decrease in self-preference bias and the highest risk of distribution collapse compared to multi-source or human data.

**Why it happens:**
LLMs have characteristic output distributions -- preferred sentence structures, vocabulary choices, explanation patterns, even punctuation habits. When one model generates all training data, these biases become the ground truth. The student model treats Opus's stylistic fingerprint as signal rather than noise. At 1.7B parameters, the model has limited capacity to separate style from substance, making it especially vulnerable.

**How to avoid:**
- Vary Opus's system prompts, temperature, and persona instructions across samples to maximize output diversity within the single-source constraint
- Use explicit style directives: "Be terse," "Respond like a senior engineer," "Explain like a textbook," "Use bullet points only" -- rotate these across the dataset
- Measure lexical diversity (type-token ratio, distinct-n) across the generated dataset; flag batches where diversity metrics drop below thresholds
- For tool-call samples specifically, vary the JSON formatting style (compact vs. pretty-printed, different key orderings) to prevent the model from memorizing one exact format
- Consider a "diversity audit" pass: cluster embeddings of generated samples and ensure even distribution across clusters

**Warning signs:**
- Generated samples all start with similar phrases ("Certainly!", "Here's how...", "Let me...")
- Low type-token ratio across the dataset (vocabulary narrowing)
- Student model outputs sound like Opus even on tasks not in the training set
- Model produces fluent-sounding but substantively wrong answers in Opus's style

**Phase to address:**
Dataset generation pipeline (Phase 1). Diversity constraints must be built into the generation prompts and validated before any training begins.

### Pitfall 2: Tool-Call Format Brittleness at 1.7B Parameters

**What goes wrong:**
Small models (under 2B parameters) struggle significantly with consistent structured output format adherence. The model learns to produce something that looks like a tool call but breaks in subtle ways: mismatched braces, wrong argument types, hallucinated function names, or correct JSON structure with nonsensical parameter values. Research on small models for function calling confirms they "struggle significantly with adhering to the given output format" and that function-calling fine-tuning "comes at a cost to certain general language capabilities, particularly in question-answering tasks."

**Why it happens:**
At 1.7B parameters, the model has limited capacity to simultaneously maintain: (a) JSON syntax rules, (b) function schema awareness, (c) argument value correctness, and (d) contextual reasoning about which tool to call. Format compliance and semantic correctness compete for the same limited parameter budget. Training on diverse JSON styles (compact, pretty-printed, varying key orders) can make this worse if the model cannot learn an invariant representation. Additionally, complex contextual nuances and ambiguous user requests require sophisticated reasoning that 1.7B models struggle to sustain alongside format requirements.

**How to avoid:**
- Standardize on ONE canonical tool-call format throughout the entire training set -- do not vary JSON whitespace, key ordering, or nesting depth
- Keep tool schemas simple: maximum 3-5 parameters per function, flat structures (no deeply nested objects)
- Include explicit "no tool needed" examples (at least 15-20% of tool-call samples) so the model learns when NOT to call tools, preventing hallucinated invocations
- Use constrained decoding at inference time (grammar-based sampling or JSON mode) as a safety net -- do not rely solely on the model to produce valid JSON
- Fine-tune on tool-call format as a separate initial phase before mixing with other tasks, so the format gets deeply encoded

**Warning signs:**
- Validation loss plateaus but tool-call format accuracy stays below 85%
- Model produces valid JSON with hallucinated function names not in the schema
- Model calls tools when it should answer directly (over-triggering)
- Correct format on simple 1-parameter calls but breaks on 3+ parameter calls

**Phase to address:**
Dataset design (Phase 1) for format standardization; evaluation framework (Phase 2) for format accuracy metrics; training (Phase 3) for potential staged fine-tuning approach.

### Pitfall 3: Catastrophic Forgetting During Multi-Task Fine-Tuning

**What goes wrong:**
Fine-tuning SmolLM2-1.7B simultaneously on tool calling, code generation, and general knowledge causes the model to lose previously learned capabilities. The model gets good at whichever task dominates the final training batches while degrading on others. This is especially severe in small models because there are fewer parameters to distribute across task-specific knowledge. Research confirms that "LoRA does not mitigate catastrophic forgetting in the context of continual learning" -- a common misconception.

**Why it happens:**
With only 1.7B parameters, the model's representational capacity is severely limited. Multi-task learning creates gradient interference: updates that improve tool-calling accuracy can overwrite weights that encoded code generation patterns. The 33/33/33 split across three domains sounds balanced but does not account for differing task difficulty -- tool calling may need disproportionately more gradient updates than general knowledge Q&A, creating implicit imbalance. Small datasets (5K total, ~1,667 per domain) amplify this because there is insufficient data to establish robust, interference-resistant representations.

**How to avoid:**
- Use task-proportional sampling with difficulty weighting, not naive 33/33/33 random shuffling -- if tool calling is hardest, it should appear more frequently in later training steps
- Implement replay-based regularization: mix in a small percentage (5-10%) of the base SmolLM2-Instruct training distribution to anchor existing capabilities
- Monitor per-task validation metrics separately throughout training, not just aggregate loss -- stop training a task early if its validation metric starts degrading
- Consider task-specific LoRA adapters (separate adapter per domain) if single-adapter multi-task proves unstable, merging later
- Evaluate the base model's zero-shot performance on all three tasks BEFORE fine-tuning as a regression baseline

**Warning signs:**
- Aggregate validation loss decreases but one task's metrics flatline or worsen
- Model suddenly loses basic language coherence (sign of severe forgetting)
- Code generation quality drops after tool-call heavy training batches
- General knowledge answers become terse or formulaic after fine-tuning

**Phase to address:**
Training strategy design (Phase 2); training execution (Phase 3). Must have per-task evaluation in place before training starts.

### Pitfall 4: The 5K Sample Ceiling -- Overfitting Disguised as Learning

**What goes wrong:**
With only ~1,667 samples per domain, the model memorizes specific examples rather than learning generalizable patterns. Training loss drops convincingly, validation loss looks fine on a held-out set from the same distribution, but the model fails on any real-world input that deviates from the narrow patterns in training data. Research shows "the smaller the dataset and the larger the epoch number, the higher the risk of overfitting" and that models scoring 95% on training distribution but failing on rephrased questions have overfit.

**Why it happens:**
5K samples is at the lower end of what research considers viable (500-5,000 for LoRA/QLoRA), but this range assumes a single task. Split across three domains, each domain gets frontier-minimum data. The model can memorize 1,667 input-output patterns with capacity to spare at 1.7B parameters, leading to a false sense of performance. Standard train/val splits from the same synthetic source do not catch this because the validation set shares the same distribution biases as the training set.

**How to avoid:**
- Hold out a genuinely out-of-distribution test set: manually crafted or sourced from a completely different LLM (not Opus) to test generalization
- Limit training to 1-3 epochs maximum; use early stopping based on the OOD test set, not the in-distribution validation set
- Use aggressive regularization: LoRA dropout of 0.1-0.15, weight decay, and moderate rank (r=16-32)
- Plan for dataset scaling: design the generation pipeline to easily produce 10K-25K samples once the initial 5K validates the approach
- Track both in-distribution and out-of-distribution metrics; if the gap exceeds 15 percentage points, overfitting is occurring

**Warning signs:**
- Training loss near zero by epoch 2
- Validation perplexity continues to look good but real-world outputs are poor
- Model reproduces exact phrases from training data verbatim
- Performance drops sharply when inputs are paraphrased or come from unexpected formats

**Phase to address:**
Evaluation framework (Phase 2) for OOD test sets; training (Phase 3) for hyperparameter tuning and early stopping; dataset scaling (Phase 4) for expanding beyond the initial 5K.

### Pitfall 5: Circular Evaluation -- Opus Generates Data, Opus Judges Quality

**What goes wrong:**
Using Claude Opus (or any single model) to both generate training data AND evaluate its quality creates a closed feedback loop. Opus rates its own outputs favorably due to self-preference bias. Research confirms that "models systematically favor outputs from their own family" and that LLM judges "consistently miss the kind of logic errors that human experts catch easily." The dataset appears high-quality by the judge's standards but contains systematic blind spots that the same judge cannot detect.

**Why it happens:**
It is natural to use the most capable available model for quality filtering. But Opus has blind spots -- specific types of errors it consistently makes or overlooks. When Opus generates a subtly wrong code snippet and then Opus evaluates it, the same reasoning flaw that produced the error will likely miss it during review. This is compounded by verbosity bias (longer, more detailed responses rated higher regardless of correctness) and prompt sensitivity in LLM-as-judge approaches.

**How to avoid:**
- Use a different model (or family of models) for quality evaluation than for generation -- even a smaller, different-architecture model catches different error classes
- Build deterministic validation for structured outputs: tool calls should be validated against schemas programmatically (not by LLM judgment); code should be syntax-checked and executed where possible
- For general knowledge, cross-reference factual claims against external sources rather than relying on LLM evaluation
- Reserve human spot-checking for a random 5-10% sample of the dataset -- small investment, catches systematic blind spots
- Use multiple evaluation dimensions scored independently (factual accuracy, format compliance, relevance, completeness) rather than a single holistic "quality" score

**Warning signs:**
- Quality filtering passes 95%+ of generated samples (suspiciously high)
- Filtered dataset shows same type-token ratios and style patterns as unfiltered
- Post-training model makes confident-sounding errors in the same domains where Opus has known weaknesses
- Evaluation scores disagree significantly when a different model family judges the same data

**Phase to address:**
Dataset quality pipeline (Phase 1) for evaluation methodology; evaluation framework (Phase 2) for deterministic validators; ongoing quality monitoring throughout.

### Pitfall 6: SmolLM2 Chat Template and Tokenizer Misalignment

**What goes wrong:**
ShareGPT format data does not automatically align with SmolLM2's specific chat template (`<|im_start|>`, `<|im_end|>` tokens). Training on data that uses a different conversation format than what the model expects produces a model that generates gibberish at turn boundaries, fails to stop generating, or ignores system prompts entirely. This is a silent failure -- training loss can look normal while the model's actual conversational behavior is broken.

**Why it happens:**
SmolLM2-Instruct uses a specific chat template with `<|im_start|>` and `<|im_end|>` special tokens. ShareGPT format uses `from`/`value` keys with `human`/`gpt` role labels. The conversion between these formats has multiple failure modes: special tokens not added to tokenizer vocabulary, padding token conflicts (pad/bos/eos set to None), chat template already present in tokenizer preventing setup_chat_format() from working, or loss computed on system/user tokens instead of only assistant tokens.

**How to avoid:**
- Verify SmolLM2's exact chat template before generating any data: `tokenizer.chat_template` shows the expected format
- Build and test the ShareGPT-to-SmolLM2 conversion pipeline on 10 samples before generating the full dataset, manually inspecting tokenized outputs
- Ensure loss masking is correctly applied: only compute loss on assistant response tokens, not user prompts or system messages
- Set `tokenizer.chat_template = None` before calling `setup_chat_format()` if the template is already present (documented workaround)
- Validate that special token IDs (bos, eos, pad) in the tokenizer match the model config and generation config

**Warning signs:**
- Model generates `<|im_end|>` tokens in the middle of responses
- Model never stops generating (missing or broken EOS token handling)
- Training loss is suspiciously low (model is predicting user tokens, not learning assistant behavior)
- Model ignores system prompts or confuses speaker roles

**Phase to address:**
Dataset format pipeline (Phase 1). This must be verified before a single sample is tokenized for training.

### Pitfall 7: Conflating Training Loss With Real Capability

**What goes wrong:**
The team watches training loss decrease, sees validation perplexity look reasonable, and concludes the fine-tuning is working. But training loss and perplexity are not reliable measures of task performance or factual accuracy. Recent research explicitly warns: "perplexity is not a reliable predictor of downstream supervised fine-tuning performance" and "a model can be confident in a completely nonsensical answer." The model can achieve low loss by learning superficial statistical patterns (common token sequences, format templates) without acquiring actual reasoning or tool-use capability.

**Why it happens:**
Loss and perplexity are convenient -- they are always available during training, require no extra infrastructure, and always go down. Building task-specific evaluation (tool-call accuracy, code execution pass rate, factual correctness) requires significant upfront investment. Teams defer building proper evaluation "until we have a model to test" and then lack the infrastructure to know whether training is actually working.

**How to avoid:**
- Build evaluation BEFORE training, not after. Define success metrics for each domain:
  - Tool calling: schema compliance rate, correct function selection rate, argument accuracy
  - Code: syntax validity, execution pass rate on test cases, functional correctness
  - General knowledge: factual accuracy against ground truth, reasoning chain validity
- Run evaluation checkpoints every N training steps, not just at the end
- Compare against the base SmolLM2-Instruct model on the same evaluations -- if the fine-tuned model is not measurably better on task metrics, the training loss improvement is meaningless
- Set minimum performance thresholds: "tool-call schema compliance must exceed 90% or we rethink the approach"

**Warning signs:**
- Team discussions focus on loss curves rather than task performance
- No evaluation infrastructure exists when training begins
- Model produces fluent text that fails functional requirements (beautiful code that does not compile, well-formatted tool calls with wrong arguments)
- Base model comparison is never run

**Phase to address:**
Evaluation framework (Phase 2) -- this is the phase most likely to be skipped under schedule pressure, and most critical to complete before training begins.

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip loss masking (train on full conversations including user turns) | Simpler training pipeline, faster setup | Model learns to predict user messages, wastes capacity, generates user-style text | Never -- always mask non-assistant tokens |
| Use Opus's default output style for all samples | Faster dataset generation, no prompt engineering | Homogeneous dataset, student mimics Opus style | Never -- diversity prompts are low-cost |
| Skip constrained decoding at inference | Simpler deployment, faster inference | Tool calls silently break in production with malformed JSON | Only if tool-call schema compliance exceeds 98% in evaluation |
| Single LoRA adapter for all three tasks | Simpler training, one model artifact | Task interference, forgetting, compromised per-task quality | Acceptable for v1 if per-task metrics remain above thresholds; revisit if any task degrades |
| Hardcode tool schemas in training data | Simpler dataset format | Model cannot adapt to new tools; requires full retrain for schema changes | Acceptable for v1 with small tool set; must be addressed before scaling |
| Use only Opus as quality judge | No additional API costs, single pipeline | Circular evaluation, systematic blind spots, false quality confidence | Only for rough initial filtering; must be supplemented with deterministic checks |

## Integration Gotchas

Common mistakes when connecting the components of this pipeline.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Opus API for data generation | Generating all 5K samples in one batch with identical prompts | Use diverse system prompts, temperature variation, and persona rotation; generate in small batches with diversity checks between batches |
| ShareGPT to SmolLM2 tokenizer | Assuming automatic format conversion works correctly | Manually verify 10+ tokenized samples; check special token alignment; validate loss masking on actual tokenized sequences |
| HuggingFace TRL/SFTTrainer | Using default SFTTrainer config without SmolLM2-specific adjustments | Verify chat_template handling, padding strategy, and that setup_chat_format() does not conflict with existing tokenizer config |
| LoRA with SmolLM2 | Applying LoRA to all layers equally | Target attention layers (q_proj, v_proj) initially; SmolLM2-1.7B uses Grouped Query Attention which affects which layers are most impactful |
| Evaluation harness | Testing only on held-out data from the same generation pipeline | Build separate OOD test sets; use programmatic validators for structured outputs; cross-model evaluation for open-ended outputs |
| HuggingFace model upload | Uploading model weights without tokenizer, config, and chat template | Always upload complete model artifacts: weights, tokenizer (with special tokens), config.json, generation_config.json, and a model card with usage examples |

## Performance Traps

Patterns that work at small scale but fail as the project grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Flat file dataset storage (single JSONL) | Works fine at 5K samples | Use HuggingFace Datasets library with streaming and sharding from the start | At 50K+ samples, loading entire file into memory becomes impractical |
| Single-GPU training without checkpointing | Fast iteration at 5K samples | Implement checkpoint saving every N steps; use gradient accumulation for effective larger batch sizes | When training on 25K+ samples or running multiple experiments; lost progress from crashes is expensive |
| Manual quality review of all samples | Feasible at 5K | Build automated quality filters and only human-review flagged outliers | At 10K+ samples, manual review becomes a bottleneck |
| Synchronous Opus API calls for generation | Simple pipeline logic | Implement async batch generation with rate limiting and retry logic | API rate limits, timeouts, and costs become prohibitive at 25K+ samples |
| Full model evaluation after each experiment | Comprehensive feedback | Implement fast proxy metrics (format compliance, length distribution) for intermediate checks; full eval only at milestones | When running 10+ hyperparameter experiments; full eval takes hours each |

## Security Mistakes

Domain-specific security and safety issues.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Opus generates training data containing real API keys, passwords, or PII in code examples | Published dataset or model reproducing sensitive data; legal/ethical liability | Explicit prompt instructions to use placeholder values; automated regex scanning of all generated data for common secret patterns |
| Model learns to execute arbitrary shell commands from tool-call training data | Deployed model generates dangerous system commands when triggered | Scope tool-call training to safe, sandboxed operations; include "refuse dangerous command" examples in training data |
| Publishing model weights without safety evaluation | Model generates harmful, biased, or toxic content inherited from Opus's training distribution | Run standard toxicity/bias benchmarks before HuggingFace release; document known limitations in model card |
| Claude API key exposure in published training scripts | API key leaked in public GitHub repo | Use environment variables exclusively; add .env to .gitignore; scrub git history before public release |

## UX Pitfalls

For downstream users of the fine-tuned Lyra model.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Model produces Opus-length responses for simple queries | Slow inference, wasted compute, users get buried in text when they want a one-liner | Train with adaptive output style: terse for code/tool calls, detailed for reasoning; include explicit "be concise" samples |
| Tool-call format works only with one inference framework | Users on different frameworks (Ollama, vLLM, llama.cpp) get broken tool calls | Test tool-call output with top 3 inference backends before release; document expected format |
| No clear documentation of model capabilities and limits | Users try tasks the 1.7B model cannot handle, blame the model | Publish detailed model card with: supported tasks, known limitations, recommended use cases, and example prompts |
| Model requires specific chat template that is not documented | Users get garbage output because they use wrong prompt format | Include chat template in tokenizer config; provide copy-paste inference code in model card; test with raw Transformers, Ollama, and vLLM |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **Dataset "complete":** Often missing diversity validation -- verify type-token ratio, embedding cluster analysis, and per-domain sample counts match targets
- [ ] **Training "converged":** Often missing task-specific evaluation -- verify tool-call accuracy, code execution pass rate, and factual correctness, not just loss curves
- [ ] **Model "works":** Often missing OOD evaluation -- verify performance on inputs NOT from the same Opus generation pipeline
- [ ] **Tool calling "functional":** Often missing edge cases -- verify behavior when no tool matches (should decline), when multiple tools match (should select correctly), and when arguments are ambiguous
- [ ] **Code generation "good":** Often missing execution testing -- verify generated code actually runs, not just that it looks syntactically plausible
- [ ] **HuggingFace release "ready":** Often missing model card, tokenizer config, chat template, generation config, license file, and inference examples
- [ ] **Evaluation "comprehensive":** Often missing base model comparison -- verify fine-tuned model actually outperforms SmolLM2-Instruct baseline on every metric claimed
- [ ] **Dataset "clean":** Often missing decontamination -- verify training data does not contain benchmark test samples that would inflate evaluation scores

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Homogeneous dataset (Pitfall 1) | MEDIUM | Regenerate 30-50% of dataset with diverse prompts; do not need to start from scratch if original samples are retained |
| Tool-call format brittleness (Pitfall 2) | HIGH | May require dataset redesign and retraining; adding constrained decoding at inference is a partial mitigation |
| Catastrophic forgetting (Pitfall 3) | MEDIUM | Retrain with replay buffer; or switch to task-specific adapters and merge; does not require new data |
| Overfitting on 5K samples (Pitfall 4) | MEDIUM | Generate more data (if pipeline exists) and retrain; reduce epochs; increase regularization |
| Circular evaluation (Pitfall 5) | LOW | Add programmatic validators and cross-model evaluation; does not require retraining, just better assessment |
| Tokenizer misalignment (Pitfall 6) | HIGH | Requires regenerating or reformatting entire dataset and retraining from scratch; all prior training is wasted |
| Trusting loss over task metrics (Pitfall 7) | LOW-MEDIUM | Build evaluation infrastructure and re-evaluate existing model; may reveal need for retraining |

## Pitfall-to-Phase Mapping

How roadmap phases should address these pitfalls.

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| Single-source homogeneity | Phase 1: Dataset Generation Pipeline | Diversity metrics (type-token ratio > 0.3, distinct-3 above baseline) computed and passing thresholds before moving to training |
| Tool-call format brittleness | Phase 1: Dataset Design + Phase 2: Eval Framework | Canonical format defined; 10+ tokenized samples manually verified; format accuracy metric implemented and reporting |
| Catastrophic forgetting | Phase 2: Training Strategy + Phase 3: Training | Per-task validation metrics tracked separately; base model baseline established; no task metric drops more than 5% below its individual peak |
| 5K sample overfitting | Phase 2: Eval Framework + Phase 3: Training | OOD test set built and integrated; gap between ID and OOD metrics below 15 percentage points |
| Circular evaluation | Phase 1: Quality Pipeline + Phase 2: Eval Framework | Programmatic validators for tool calls and code; at least one non-Opus evaluation signal in pipeline |
| Tokenizer misalignment | Phase 1: Format Pipeline | Manual inspection of 10+ tokenized samples showing correct special tokens, loss masking, and turn boundaries before any training |
| Trusting loss over task metrics | Phase 2: Eval Framework | Task-specific metrics defined, implemented, and producing results before first training run launches |
