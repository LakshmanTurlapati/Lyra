# Phase 09.1: Tool-Call Format Regression Fix - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the tool-call-format eval regression (0.41 → 0.19) by resolving the chat template persistence bug, scaling training data to 25K samples, retraining, and re-evaluating to confirm Lyra outperforms the base model.

</domain>

<decisions>
## Implementation Decisions

### Root Cause
- **D-01:** The regression is caused by a chat template persistence bug, NOT a data quality issue. `train.py:435-455` sets a custom chat template during training that converts `tool_calls` fields to `<tool_call>JSON</tool_call>` XML format. But `models/lyra-merged/tokenizer_config.json` does NOT include the `chat_template` field after merge. During eval, the tokenizer falls back to SmolLM2's default template which has no tool-call handling — creating a distribution mismatch.
- **D-02:** The base model scores higher (0.41) because it does pure in-context mimicry of `<tool_call>` tags. Lyra's fine-tuned patterns are template-specific and don't transfer when the wrong template is applied during eval.

### Fix Strategy (3-stage)
- **D-03:** Stage 1 — Quick eval: Fix eval_inference.py to explicitly load the custom chat template from the model directory, re-run eval on existing Lyra model. This validates the diagnosis before investing in retraining.
- **D-04:** Stage 2 — Fix template persistence: Modify train.py's merge step to explicitly write `chat_template` into `tokenizer_config.json` after `merge_and_unload()`. This is the root cause fix — single source of truth in the training pipeline.
- **D-05:** Stage 3 — Scale data + retrain: Generate ~21K additional training samples (to reach 25K total), retrain with the template fix baked in, then re-evaluate.

### Data Scaling
- **D-06:** Keep current domain ratios when scaling: ~68% tool-calling (~17K), ~16.5% code (~4K), ~15.4% knowledge (~4K). Scale all domains proportionally.
- **D-07:** Generation uses existing template-based scripts (gen_tool_calls.py, gen_code.py, gen_knowledge.py) with more batches/seeds. No API calls needed — local generation only.
- **D-08:** After scaling, run the full curation pipeline (dedup, quality scoring, format validation) on all 25K samples before training.

### Success Criteria
- **D-09:** Minimum acceptable tool-call-format score: must beat base model (>0.41). This is the whole point of fine-tuning.
- **D-10:** Knowledge benchmarks (MMLU, ARC, HellaSwag) should not degrade more than 5% from base model scores.

### Execution Constraints
- **D-11:** All model evaluations MUST run sequentially — parallel eval crashed 64GB machine (80GB+ usage). Never run two model-loading processes simultaneously on MPS.
- **D-12:** Use --limit 100 for lm-eval benchmarks (same as Phase 9 baseline for fair comparison).

### Claude's Discretion
- Eval_inference.py template loading implementation details
- Whether to use chat_template.jinja file or inline string in tokenizer_config.json
- Batch count distribution across categories when generating additional samples
- LoRA hyperparameter adjustments for larger dataset (if any needed)

</decisions>
