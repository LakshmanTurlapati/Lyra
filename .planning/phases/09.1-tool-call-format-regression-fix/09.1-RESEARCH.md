# Phase 09.1: Tool-Call Format Regression Fix - Research

**Researched:** 2026-04-21
**Domain:** LLM eval diagnostics, chat template persistence, dataset scaling, LoRA hyperparameters
**Confidence:** HIGH (codebase verified), MEDIUM (scaling recommendations)

## Summary

Phase 09.1 addresses the tool-call-format score regression from 0.41 (base) to 0.19 (Lyra). The CONTEXT.md diagnosis attributes the regression to a chat template persistence bug where `tokenizer_config.json` lacks the `chat_template` field. However, hands-on verification reveals a critical nuance: transformers 5.5.4 saves the template to `chat_template.jinja` (not `tokenizer_config.json`) and `from_pretrained` correctly loads it from that file. The template IS being loaded during eval and produces token-identical prompts to the base model template.

The real regression cause is more likely a combination of: (1) the eval methodology measuring something unintended -- ALL 123 tool-calling test samples have text content as their expected output, not tool calls, so the metric is really "how often does the model hallucinate tool-call format when it should produce text," and (2) the model's learned patterns during fine-tuning genuinely changing its tool-call generation behavior. The base model scores 0.41 because it in-context mimics `<tool_call>` tags ~41% of the time; Lyra scoring 0.19 may actually indicate less hallucination, not worse performance.

Despite this finding, the CONTEXT.md 3-stage strategy is still sound: Stage 1 (quick eval with explicit template) will empirically validate whether the template matters. If the score stays at 0.19, the diagnosis is confirmed wrong and the real issue is model quality. The data scaling (Stage 3) is the actual fix -- more data will improve the model's ability to generate tool calls when appropriate. The template persistence fix (Stage 2) is low-cost insurance that prevents future confusion.

**Primary recommendation:** Execute Stage 1 quickly as a diagnostic. If the score does NOT improve, pivot Stage 2 to focus on eval methodology (consider adding test samples where tool_calls IS the expected output). Stage 3 data scaling proceeds regardless -- 25K samples will improve model quality.

## Critical Research Finding: Template Diagnosis

**IMPORTANT: The CONTEXT.md diagnosis (D-01) may be incorrect.**

### Verified Facts

1. **`chat_template.jinja` exists and is loaded.** The file `models/lyra-merged/chat_template.jinja` was created at 2026-04-21 01:42:29, BEFORE the eval ran at 21:27:14. Transformers 5.5.4 `AutoTokenizer.from_pretrained('models/lyra-merged')` correctly loads this file and sets `tokenizer.chat_template`. [VERIFIED: tested in venv Python with transformers 5.5.4]

2. **Templates produce identical tokens.** Both the base SmolLM2 template and Lyra's custom template (with `{% generation %}` markers) produce bit-identical token sequences for the same input messages. The `{% generation %}` markers are TRL-specific extensions that have zero effect during inference -- they are silently ignored by the Jinja2 renderer. [VERIFIED: compared token IDs side-by-side in venv Python]

3. **`tokenizer_config.json` does NOT have `chat_template` field.** This is true but irrelevant for transformers 5.5.4, which loads templates from `chat_template.jinja` files with higher priority than `tokenizer_config.json` inline strings. [VERIFIED: transformers docs via Context7 confirm `.jinja` is the standard persistence mechanism]

4. **The eval prompts are identical** whether using the base template or Lyra's template. The normalization step in `_normalize_messages_for_template()` converts `tool_calls` to `<tool_call>` content BEFORE applying the template, making the template's own tool_calls handling redundant during eval. [VERIFIED: traced eval flow in Python]

### Eval Methodology Concern

**All 123 tool-calling test samples expect TEXT output, not tool calls.**

Analysis of the test split:
- 123 tool-calling test samples total [VERIFIED: dataset inspection]
- 0 samples have tool_calls as the last assistant turn [VERIFIED: dataset inspection]
- 22 are "no-tool edge cases" where the model should NOT call tools [VERIFIED: dataset inspection]
- 101 have prompts ending with a tool result; the expected output is a text summary [VERIFIED: dataset inspection]

The `check_tool_call_format` metric measures: "of all tool-calling test samples, how many model outputs contain valid `<tool_call>JSON</tool_call>` format?" Since the expected output for ALL samples is text (not tool calls), this metric is measuring hallucinated tool calls.

- **Base model: 0.41** = hallucinates tool_call format 41% of the time when it should produce text
- **Lyra: 0.19** = hallucinates tool_call format 19% of the time

**This could be an improvement, not a regression.** The planner should consider adding test samples where tool_calls IS the expected output (single-turn prompts that end with `[system, user]` and no prior tool interaction).

**Risk level if CONTEXT.md diagnosis is wrong:** LOW for Stage 1 (quick eval is cheap -- it validates the diagnosis). MEDIUM for Stage 3 (data scaling improves model quality regardless). The 3-stage strategy is still valid; only the interpretation of Stage 1 results may differ.

### Pattern 1: Template Persistence Fix

**What:** Ensure the custom chat template (with tool_calls handling) persists in the merged model directory.

**Recommendation:** Use the `.jinja` file approach (already working in transformers 5.5.4). The current `tokenizer.save_pretrained()` already saves to `chat_template.jinja`. No code change needed for template persistence itself.

**However**, for robustness and compatibility with older transformers versions or third-party tools that read `tokenizer_config.json`, also write the template inline. This is the "belt and suspenders" approach:

```python
# Source: verified behavior in transformers 5.5.4
# In train.py merge step, AFTER tokenizer.save_pretrained():

import json

# Also write to tokenizer_config.json for compatibility
config_path = merged_dir / "tokenizer_config.json"
config = json.loads(config_path.read_text())
# Strip TRL-specific {% generation %} markers for inference template
inference_template = tokenizer.chat_template.replace("{% generation %}", "").replace("{% endgeneration %}", "")
config["chat_template"] = inference_template
config_path.write_text(json.dumps(config, indent=2))
```

[VERIFIED: tokenizer.save_pretrained saves .jinja but NOT inline in config.json]

### Anti-Patterns to Avoid

- **Parallel eval on MPS:** Never run two model-loading processes simultaneously. MPS memory management cannot handle 2x 3.4GB models. [VERIFIED: Phase 9 experience]
- **Generating without curation:** Never skip the curation pipeline when adding new data. Template-based generation produces duplicates that dedup catches. [VERIFIED: 73.6% retention rate for tool-calling data]
- **Reusing exact seed ranges:** When generating additional batches, start batch numbers AFTER existing ones. Seeds are derived from batch numbers (batch*100 + offset). Reusing batch numbers = duplicate data. [VERIFIED: codebase pattern in generate_tool_data.py]

## Common Pitfalls

### Pitfall 1: Template-Data Mismatch at Scale
**What goes wrong:** At 7x scale, template-based generation produces many near-duplicates that pass individual quality checks but create a homogeneous training set. The model learns to regurgitate templates rather than generalize.
**Why it happens:** Finite query templates (113 total for tool-calling) with finite value pools (12 cities, 10 topics, etc.) have limited combinatorial diversity.
**How to avoid:** Monitor curation retention rate. If it drops below 50% at scale, expand query template pools and value pools before generating more batches. Consider varying system prompts across batches for additional diversity.
**Warning signs:** Dedup stage removing >40% of samples; model outputs at eval time look templated; training loss drops very quickly (overfitting).

### Pitfall 2: Training Time Explosion
**What goes wrong:** 25K dataset with current hyperparameters takes 3.5-7 hours on MPS (vs 30-60 min for 3.6K). A single failed training run wastes hours.
**Why it happens:** 6.9x more data = 6.9x more training steps at same batch size/epochs.
**How to avoid:** Reduce epochs from 3 to 2 for initial retrain (more data per epoch compensates). Reduce learning rate from 2e-4 to 1e-4 (larger dataset needs slower convergence). Use --max-steps for a quick smoke test first.
**Warning signs:** Training loss not converging after 1000 steps; MPS kernel crashes mid-training.

### Pitfall 3: Eval Metric Misinterpretation
**What goes wrong:** The tool-call-format metric measures "how often the model produces `<tool_call>` format in its output" across ALL tool-calling test samples. But not all test samples SHOULD produce tool calls -- most expect text summaries.
**Why it happens:** The test split was designed for multi-turn conversations where the label is always a text summary after seeing tool results. There are no test samples where the expected output IS a tool call.
**How to avoid:** Consider adding test samples where the last assistant turn should be a tool call (single-turn: `[system, user]` -> expected: tool_call). This gives a more meaningful metric.
**Warning signs:** Score improvement after retrain that doesn't match qualitative inspection of model outputs.

### Pitfall 4: Memory Exhaustion During Eval
**What goes wrong:** Running lm-eval and custom eval in parallel or in rapid succession causes MPS to run out of memory (80GB+ peak).
**Why it happens:** MPS memory pool does not immediately release after model deletion. `gc.collect()` helps but is not sufficient if models are loaded back-to-back.
**How to avoid:** Run evals as separate processes (not within the same Python session). Add a short delay between model-loading processes. Sequential execution only.
**Warning signs:** Process killed with signal 9; MPS runtime errors about memory allocation.

### Pitfall 5: Curation Config Mismatch at Scale
**What goes wrong:** Running curation on mixed old+new data with per-domain configs may produce different results than the original curation (old data could get filtered out).
**Why it happens:** The dedup algorithm compares ALL samples against each other. Adding new similar samples can cause previously-passing old samples to become duplicates.
**How to avoid:** Run curation on the ENTIRE raw dataset (old + new batches together), not incrementally. Accept that the final curated count may differ from (old curated + new curated).
**Warning signs:** Final curated count significantly less than expected; known-good samples from Phase 4 disappearing.

### Fix 2: Explicit Template Loading in eval_inference.py

```python
# Source: eval_inference.py lines 125-132

def _do_load_model_and_tokenizer(model_path: str, device: str):
    import torch
    from pathlib import Path
    from transformers import AutoModelForCausalLM, AutoTokenizer

    logger.info("Loading model from %s onto %s", model_path, device)
    model = AutoModelForCausalLM.from_pretrained(
        model_path, torch_dtype=torch.float32
    ).to(device)
    tokenizer = AutoTokenizer.from_pretrained(model_path)
    
    # Verify template loaded (defensive check)
    if tokenizer.chat_template is None:
        jinja_path = Path(model_path) / "chat_template.jinja"
        if jinja_path.exists():
            tokenizer.chat_template = jinja_path.read_text()
            logger.warning("Loaded chat_template from .jinja fallback")
        else:
            logger.warning("No chat_template found -- using model default")
    
    return model, tokenizer
```

## Open Questions

1. **Is the regression actually an improvement?**
   - What we know: All 123 test samples expect text output, not tool calls. Lower score = fewer hallucinated tool calls.
   - What's unclear: Whether reducing hallucinated tool calls correlates with better actual tool-calling ability.
   - Recommendation: Stage 1 will empirically test this. If score stays at 0.19 with explicit template, the regression is model behavior, not template loading. Consider adding 20-30 test samples where tool_calls IS the expected output.

2. **Will template-based generation produce enough diversity at 17K scale?**
   - What we know: ~9000 estimated unique combos for tool-calling templates. Current 2464 curated = 27% of theoretical max.
   - What's unclear: How aggressive dedup will be with 5-7x more data from the same template pools.
   - Recommendation: Generate a pilot batch of 100 batches, run dedup, measure retention rate before committing to full 400-batch generation.

3. **Do hyperparameters need adjustment for 7x data?**
   - What we know: Standard practice is lower LR and fewer epochs with more data. But SmolLM2-1.7B is a small model.
   - What's unclear: Whether the model's capacity is sufficient for 25K diverse samples without overfitting or underfitting.
   - Recommendation: Use --max-steps 100 for a quick validation before full training. Check loss convergence.
