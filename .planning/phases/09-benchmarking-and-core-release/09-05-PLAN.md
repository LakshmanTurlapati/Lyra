### Usage

Load and run inference with the fine-tuned model:

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model_path = "models/lyra-merged"  # or HuggingFace Hub path once published

tokenizer = AutoTokenizer.from_pretrained(model_path)
model = AutoModelForCausalLM.from_pretrained(

).to("mps")  # or "cpu"

messages = [

prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,

inputs = tokenizer(prompt, return_tensors="pt").to("mps")

with torch.no_grad():
    outputs = model.generate(**inputs, max_new_tokens=256, do_sample=False)

new_tokens = outputs[0][inputs["input_ids"].shape[1]:]
print(tokenizer.decode(new_tokens, skip_special_tokens=True))

# Lyra Training Dataset

## Dataset Description

Synthetic instruction-tuning dataset for fine-tuning SmolLM2-1.7B on tool calling, code generation, and general knowledge tasks. All training data is generated by Claude Opus using the Anthropic Message Batches API with a multi-stage quality pipeline (deduplication, format validation, quality scoring).

The dataset uses TRL-native format (`messages`/`role`/`content`) with SmolLM2-specific `<tool_call>` XML wrapper for tool calls. It is validated against the SmolLM2-1.7B tokenizer and chat template before inclusion.

## Creation Methodology

1. **Template-based generation**: Domain-specific prompt templates covering 64 tool schemas (tool-calling), 5 programming languages (code), and 192 unique topic pairs (knowledge)
2. **Batch generation**: Claude Opus generates all samples via the Message Batches API (50% cost reduction)
3. **Quality pipeline**: Each sample passes through n-gram Jaccard deduplication, Pydantic format validation, and heuristic quality scoring (response length, prose ratio, vocabulary richness, meta-commentary detection)
4. **Tokenizer validation**: Every sample validated against SmolLM2-1.7B's tokenizer and chat template (token count, EOS presence, no default system prompt injection)
5. **Stratified assembly**: Domains merged with 90/5/5 train/validation/test split maintaining domain proportions

## Dataset Statistics

| Split | Samples | Tool-Calling | Code | Knowledge |
|-------|---------|-------------|------|-----------|
| Train | 3,262 | ~2,218 (68%) | ~542 (17%) | ~502 (15%) |
| Validation | 181 | 123 (68%) | 30 (17%) | 28 (15%) |
| Test | 181 | 123 (68%) | 30 (17%) | 28 (15%) |
| **Total** | **3,624** | **~2,464** | **~602** | **~558** |

### Code Subcategories

### Knowledge Subcategories

|-------------|-------|
| Factual Q&A | ~185 |

## Limitations

- **Synthetic data**: All samples generated by Claude Opus; may not capture natural human conversational patterns or edge cases outside the template design
- **Format-specific**: Tool call format uses SmolLM2-specific `<tool_call>XML</tool_call>` wrapper; not directly compatible with OpenAI function calling format
- **English only**: No multilingual samples; performance on non-English tasks is not evaluated
- **Bounded topic coverage**: Knowledge domain uses a 192-topic pool; coverage is intentionally narrow for quality over breadth
- **Scale**: 3,624 samples is small by industry standards; capability gains are expected to be meaningful but bounded
