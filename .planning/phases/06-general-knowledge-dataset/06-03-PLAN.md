**Step 4: If retention is below 50% (below ~1,667 samples), tune the knowledge dedup config.**

Based on Phase 5 lessons, the likely cause is dedup being too aggressive at 0.7 threshold. The mitigation path (in order of preference):

a) If retention is 40-49%: Raise dedup_threshold to 0.8 in pipeline.yaml for knowledge domain. Re-run pipeline.

b) If retention is 30-39%: Raise dedup_threshold to 0.85 AND switch dedup_scope to "user-response" (excludes shared system prompts from similarity, same as code domain fix in Phase 5). Re-run pipeline.

c) If retention is below 30%: Raise dedup_threshold to 0.9 with user-response scope. Re-run pipeline.

When modifying pipeline.yaml, add the override under the knowledge domain section:
```yaml
knowledge:
  min_response_chars: 100
  dedup_threshold: 0.8  # Raised from default 0.7 if needed
  dedup_scope: "user-response"  # Added if needed
  style:
    min_tokens: 200
    require_reasoning_markers: true
    min_prose_ratio: 0.7
```
