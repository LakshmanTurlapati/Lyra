Per TOOL-02, each multi-turn conversation MUST follow the pattern:
```
system -> user -> assistant(tool_calls) -> tool -> assistant(content) -> user(follow-up) -> assistant(tool_calls) -> tool -> assistant(content)
```

That is: minimum 2 tool call rounds. Each round is: assistant calls tool, gets result, summarizes. Then user asks a follow-up question that builds on the previous result, triggering another tool call.

Structural requirements:
- 2-3 tools defined in `tools` array (max 3 to conserve tokens -- budget is ~1750)
- 2-3 full tool call rounds per conversation (user -> assistant(tool_calls) -> tool -> assistant)
- Follow-up questions must reference information from the previous tool response
- Final message is always assistant with content (no dangling tool_calls)
- Total token count must stay under 2048 (validate with validate_tokenizer.py)

Topic variety: Use diverse tool combinations from the schema pool. Examples:
- Search packages then get details on a specific result
- Query database then look up a related record
- Check weather then get directions to a destination
- List files then read a specific file's contents

Per D-05: ~25% edge cases (~124 samples total):
- No-tool-needed: follow-up is a simple question the assistant answers directly
- Error handling: second tool call fails, assistant recovers and answers from first result
- Ambiguous: user's follow-up could be answered from memory or by calling another tool

Per TOOL-04, MCP conversations follow the discovery-then-invoke pattern:
```
system -> user(request)
  -> assistant(tool_calls: [mcp_list_servers]) -> tool(server list)
  -> assistant(tool_calls: [mcp_list_tools]) -> tool(tool list)
  -> assistant(tool_calls: [discovered_tool]) -> tool(result)
  -> assistant(content: final answer)
```

This is a 3-round multi-turn pattern (list_servers, list_tools, invoke). The key difference from regular multi-turn is that each step discovers what to do next.

CRITICAL TOKEN BUDGET: MCP has the tightest budget (~1720 tokens for content). Keep responses concise:
- Server list responses: short JSON arrays with 2-3 servers
- Tool list responses: 2-3 tools with brief descriptions
- Discovered tool responses: concise result objects
- Assistant summaries: 1-2 sentences per intermediate step, 2-3 sentences for final answer

MCP-specific tools from schema pool:
- `mcp_list_servers`: no params, returns server list
- `mcp_list_tools`: takes `server` param, returns tool list
- Discovered tools: drawn from the developer/everyday pool but presented as "discovered" via MCP

Topic variety for MCP scenarios:
- "Check our monitoring system" -> discover monitoring server -> list health tools -> invoke
- "Query the database" -> discover data server -> list query tools -> invoke
- "Send a notification" -> discover notification server -> list messaging tools -> invoke
- "Deploy latest changes" -> discover CI/CD server -> list deployment tools -> invoke
- "Search the knowledge base" -> discover search server -> list search tools -> invoke

Vary the MCP server names: monitoring, database, notifications, ci-cd, search, analytics, logging, auth, storage, cache.

Per D-05: ~25% edge cases (~83 samples):
- No-tool-needed: "What is MCP?" -- assistant explains without discovering servers
- Error handling: mcp_list_servers returns empty list or error, assistant explains no servers available
- Error handling: mcp_list_tools returns empty for a server, assistant tries another server
- Ambiguous: user request could be handled by multiple servers
