# Phase 09.2: Tool-Call Regression Diagnosis - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Diagnose why the Phase 09.1 retrained Lyra v2 scored 0.0634 on tool-call-format (below base 0.4065 and far below the >0.41 target), then execute the correct fix — which may be a full retrain, a targeted patch, or a revert to the Phase 9 checkpoint. Phase 09.2 closes when either the fine-tuned model beats the base on tool-call-format (>0.41) with MMLU/ARC/HellaSwag within 5% of base, or a documented revert-to-Phase-9 release is in place.

</domain>

<decisions>
## Implementation Decisions

### Scope & Fallback Strategy

- **D-01 (Scope):** Phase 09.2 is a full diagnose-and-fix phase. It does not stop at a root-cause report — it carries through to the corrective action (retrain, patch, or revert). Matches the ambition of 09.1 but grounded in evidence rather than a single hypothesis.
- **D-02 (Fallback is valid):** If diagnosis shows no cheap recovery from the v2 model state, reverting to the Phase 9 release checkpoint (tool-call-format 0.4065) is an acceptable shipping outcome. The plan must explicitly carry a revert-path task so it is actionable rather than hypothetical.

### Evidence to Collect (diagnosis feeds the fix decision)

- **D-03 (Raw generations):** Read `results/lyra_v2_custom.json` and hand-inspect a representative sample of tool-call-format failures. Categorize each as: empty output, plain-text answer (no `<tool_call>`), malformed XML, wrong function signature, or TRL generation marker leakage (`{% generation %}` or similar). This alone usually reveals the failure mode.
- **D-04 (Template parity check):** Tokenize an identical tool-call prompt with (a) the chat template that `scripts/train.py` applies in-memory during SFT and (b) the `chat_template` string persisted in `models/lyra-merged/tokenizer_config.json`. Produce a token-ID diff. Any divergence is a smoking gun for train/eval mismatch.
- **D-05 (Fresh inference probe):** Build a small standalone Python probe (not `eval_inference.py`) that loads `models/lyra-merged`, renders 5–10 tool-call prompts using the persisted template, and prints raw model output. Removes eval-harness confounds; verifies the model can produce *any* tool-call output under controlled conditions.
- **D-06 (Training-run review):** Revisit `09.1-05-SUMMARY.md` loss curve (epoch 1: 0.1613 → epoch 2: 0.1384) against Phase 8 training loss curves. Flag catastrophic-forgetting signatures or suspiciously low loss that could indicate the model learned a degenerate shortcut.

### Eval Methodology Sanity Check (guard against chasing a ghost)

- **D-07 (Re-verify eval first):** Before touching the model, re-run `scripts/eval_inference.py` and `scripts/eval_runner.py` on the base SmolLM2-1.7B and confirm the base scores reproduce (`tool-call-format ≈ 0.4065`, and knowledge benchmarks within noise). If the eval pipeline has drifted since Phase 9, the 0.0634 is partially or fully an eval bug and the fix target shifts.
- **D-08 (Pre-09.1 Lyra is lost):** The old `models/lyra-merged/` was overwritten by Plan 05's merge step. No pre-09.1 Lyra artifacts exist. Diagnosis proceeds from (v2 merged model, base model) as the only two comparison points. Do not spend a retrain cycle reconstructing the old Lyra.

### Exit Criteria

- **D-09 (Success bar):** Phase 09.2 succeeds when the final shipped model beats the base on tool-call-format (`pass@1 > 0.41`) AND satisfies the D-10 guard from Phase 09.1 (MMLU, ARC-Challenge, HellaSwag each within 5% of base). "Final shipped model" can be a retrained Lyra OR a documented revert to the Phase 9 checkpoint — both are valid exits if they clear the bar.
- **D-10 (Retrain budget):** No cap on retrain attempts. Evidence from D-03 through D-07 must justify each retrain's hypothesis; Claude should not kick off a retrain without a changed hypothesis on the table. A retrain takes ~3.5h on MPS, so each attempt has real wall-clock cost.

### Global Execution Constraints (inherited, do not re-decide)

- **D-11 (Sequential evals):** All model-loading processes run sequentially. Never run two evals or an eval + training concurrently. Prior incident: parallel evals crashed a 64GB machine (80GB+ usage). (Carried from 09.1 D-11 and reinforced by user memory.)
- **D-12 (Eval limits):** Knowledge benchmarks use `--limit 100`, consistent with Phase 9 baseline and Phase 09.1 for fair comparison.

### Claude's Discretion

- Exact structure of the fresh inference probe script (one-off diagnostic, not a long-lived utility).
- Specific hyperparameter changes to test in retrain (lr, epochs, sample weighting) — must flow from D-03/D-04/D-06 evidence; planner decides after research.
- Whether to introduce domain rebalancing (e.g., oversampling the 649/651 code/knowledge samples) vs accepting the 90.2% tool-calling skew.
- Fix sequencing: whether to try a cheap eval-side patch first before a retrain when diagnosis is ambiguous.

</decisions>

<specifics>
## Specific Ideas

- Sequencing: D-07 (eval sanity) before D-03/D-04/D-05 (model-side evidence). A broken eval invalidates all model-side conclusions, so we shouldn't invest in deep model diagnosis before the eval is known-good.
- If D-03 raw generations show TRL generation markers (`{% generation %}`) leaking into outputs, that is a strong signal that `train.py`'s in-memory template during SFT has those markers while the persisted template does not — token embeddings would be drifted on those marker tokens.
- The 90.2% tool-calling skew in the training set (D-03 09.1-04 Plan 04) is suspicious for a format regression: the model saw *more* tool-call examples than before, yet produces *worse* format. Suggests the failure is not a data-starvation issue.
- Keep diagnostic scripts (probe_generations.py, diff_templates.py) out of the main pipeline — one-off, commit to the phase directory or delete after the phase if they're not worth keeping.

</specifics>
