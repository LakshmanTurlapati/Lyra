       - Title: "Lyra ShareGPT Format Specification"
       - Section "Overview": TRL-native conversational format using messages/role/content keys (per D-01). NOT classic ShareGPT from/value format. State that this is the single source of truth for all data generation.
       - Section "Format Structure": Each JSONL line is a JSON object with:
         - `messages` (required): list of message objects with `role` (string: system|user|assistant|tool), `content` (string or null), `tool_calls` (optional list), `name` (optional string, required for tool role)
         - `tools` (optional): list of tool schema objects with `type` ("function"), `function` (object with `name`, `description`, `parameters`)
       - Section "Role Ordering Rules" (per D-02, all 9 rules from RESEARCH.md lines 668-679):
         1. First message MUST be system role
         2. After system, messages alternate user/assistant (with tool interludes)
         3. tool messages MUST immediately follow assistant with tool_calls
         4. Multiple tool messages can follow single assistant with multiple tool_calls (parallel)
         5. assistant with tool_calls must be followed by exactly one tool message per tool call
         6. Final message SHOULD be assistant role
         7. assistant with tool_calls typically has content null or empty
         8. tool messages MUST have name field matching called tool
         9. tool content should be string representation of return value
       - Section "Tool Call Patterns" (per D-02, all 5 patterns):
         - Single function call: JSON example with one tool_calls entry
         - Multi-turn with results: JSON example showing user -> assistant[tool_calls] -> tool -> assistant cycle
         - Parallel execution: JSON example with multiple tool_calls in one assistant message, followed by matching tool messages
         - MCP-style: JSON example showing server discovery + tool listing + invocation + result
         - CLI/shell commands: JSON example with CLI tool (run_command) and shell output
       - Section "Token Budget": Maximum 2048 tokens AFTER apply_chat_template(tokenize=True) per D-03. Natural distribution per D-04.
       - Section "Storage": JSONL format, one conversation per line, stored in datasets/ per D-09 (datasets/tool-calling/, datasets/code/, datasets/knowledge/)
       - Section "Important Notes": PAD/EOS token collision (id=2 for both), default system prompt injection if system role missing, mapping from legacy terminology (function_call role -> assistant with tool_calls, observation role -> tool role)

    6. Create scripts/validate_format.py with the following exact structure (per D-06, D-07, D-08):
       - Pydantic models at module level:
         - `class FunctionDef(BaseModel)`: fields `name: str`, `description: str`, `parameters: dict`
         - `class ToolSchema(BaseModel)`: fields `type: str = "function"`, `function: FunctionDef`
         - `class ToolCallFunction(BaseModel)`: fields `name: str`, `arguments: dict`
         - `class ToolCall(BaseModel)`: fields `type: str = "function"`, `function: ToolCallFunction`
         - `class Message(BaseModel)`: fields `role: str`, `content: Optional[str] = None`, `tool_calls: Optional[list[ToolCall]] = None`, `name: Optional[str] = None`
         - `class Conversation(BaseModel)`: fields `messages: list[Message]`, `tools: Optional[list[ToolSchema]] = None`
           - `@model_validator(mode='after')` method `validate_structure` that enforces all 9 role ordering rules:
             - Reject empty messages (ValueError: "Empty conversation")
             - First message must be system (ValueError: "First message must be system role")
             - Valid roles are only: system, user, assistant, tool (ValueError: "Invalid role '{role}' at index {i}")
             - tool messages must follow assistant or tool (ValueError: "Tool message at index {i} must follow assistant or tool")
             - tool messages must have name field (ValueError: "Tool message at index {i} missing 'name' field")
             - If tools column present, tool_calls must reference defined tool names (ValueError: "Tool call '{name}' not in defined tools: {defined_names}")
             - Count tool_calls in preceding assistant message and verify matching count of subsequent tool messages (ValueError: "Expected {expected} tool messages after assistant at index {i}, got {actual}")
       - Function `validate_file(path: Path) -> dict` that reads JSONL line by line, validates each with Conversation.model_validate(), returns `{"total": int, "valid": int, "invalid": int, "errors": [{"line": int, "error": str}]}`
       - CLI entry point with argparse: `python scripts/validate_format.py <jsonl_file>` that prints validation results as JSON to stdout with exit code 0 if all valid, 1 if any invalid
       - No emojis in any log output or error messages

    All imports: `json`, `sys`, `argparse` from stdlib; `Path` from pathlib; `BaseModel`, `model_validator` from pydantic; `Optional` from typing.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JSONL file input -> validation script | Untrusted JSONL data from disk parsed by Pydantic |
| User-supplied file paths -> validate_format.py CLI | File path argument from command line |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | validate_format.py JSONL parsing | mitigate | Pydantic model_validate() rejects unexpected fields; json.loads() handles encoding safely; no eval() or exec() on input data |
| T-01-02 | Tampering | validate_format.py file path argument | mitigate | Use pathlib.Path for path handling; argparse validates argument is provided; open() will raise FileNotFoundError for invalid paths |
| T-01-03 | Information Disclosure | Error messages in validation output | accept | Error messages contain field names and values from input data; acceptable for local CLI tool with no network exposure |
| T-01-04 | Denial of Service | Extremely large JSONL files | accept | Phase 1 operates on small files (10-50 samples); streaming line-by-line read prevents full file load into memory |
</threat_model>
