# Lyra

An open-source dataset curation and model fine-tuning project that distills Claude Opus reasoning quality into training data for SmolLM2-1.7B.

## What This Is

Small language models are fast and locally deployable, but they lack capability in structured tasks like tool calling and code generation. SmolLM2-1.7B is HuggingFace's largest SmolLM variant -- its base instruction-following ability is limited, but its size makes it ideal for local deployment. The hypothesis is that carefully curated, high-quality training data can close a significant portion of the capability gap between this tiny model and much larger ones, especially in structured tasks where format consistency matters more than raw parameter count.

Named in the musical tradition of Haiku, Sonnet, and Opus -- Lyra is a small constellation that produces clear, harmonious output. The goal is to curate Opus-quality training data that makes a 1.7B parameter model practically useful for day-to-day development tasks: tool calls, quick code, and general reasoning. All training data is generated by Claude Opus with no human annotation in v1, starting with approximately 5,000 samples and growing iteratively based on evaluation results.

## Core Features

- **Tool Calling** -- JSON-structured function calls, multi-turn tool use with results handling, parallel execution, MCP-style server discovery and invocation, and CLI/shell command generation. Approximately one-third of the training data focuses on teaching the model when and how to use tools, including when not to call a tool.

- **Code Generation** -- Quick utility functions across Python, JavaScript, TypeScript, Go, and Rust. File operations, debugging samples that identify bugs, explain root causes, and provide corrected code. Responses are concise and practical -- code-first with minimal prose.

- **General Knowledge** -- Reasoning chains with explicit step-by-step chain-of-thought, factual Q&A spanning science, history, math, and technology, and teaching explanations that adapt depth to topic complexity. The training data teaches the model to be terse when brevity serves the user and detailed when the topic demands it.

All training data is validated against SmolLM2-1.7B's tokenizer and chat template before inclusion. The data format is TRL-native (`messages`/`role`/`content` with structured `tool_calls`), not the legacy ShareGPT `from`/`value` format, so it works directly with HuggingFace's SFTTrainer with no conversion step.

## Architecture

The pipeline follows a format-first discipline. The data format specification and validation infrastructure are established before any data generation at scale. Every conversation is validated in two stages: first against a Pydantic schema that enforces role ordering, tool call consistency, and structural correctness, then against the SmolLM2-1.7B tokenizer to verify token count limits, EOS token presence, chat template markers, and absence of default system prompt injection.

Each stage of the pipeline reads and writes JSONL independently. Domain datasets (tool calling, code, general knowledge) are generated and curated separately, then assembled into a final dataset with stratified train/validation/test splits and balanced domain representation. The evaluation framework is built before training begins, so every training run produces actionable signal from day one.

Training uses QLoRA (4-bit quantization) via Unsloth and TRL's SFTTrainer. This fits comfortably on consumer GPUs with 8GB+ VRAM. QLoRA produces small adapter files (~50-100MB) that can be merged into the base model for deployment. The base model is SmolLM2-1.7B-Instruct with an 8,192 token context window, though training samples target 2,048 tokens to stay within the model's effective range.

## Setup

Python 3.10+ required.

```
git clone https://github.com/LakshmanTurlapati/Lyra.git
cd Lyra
pip install -r requirements.txt
```

The `requirements.txt` currently contains Phase 1 dependencies only (pydantic, transformers, pyyaml, pytest). Additional dependencies for fine-tuning (unsloth, trl, peft, bitsandbytes) and evaluation (lm-eval) will be added in later phases.

```
# Validate a JSONL dataset against the format specification
python -m scripts.validate_format datasets/tool-calling/samples.jsonl

# Validate tokenizer alignment against SmolLM2-1.7B (downloads model on first run)
python -m scripts.validate_tokenizer datasets/tool-calling/samples.jsonl

# Generate sample conversations for pipeline testing
python -m scripts.generate_sample

# Run tests
pytest
```

## Current Status

Lyra has working data generation, curation, dataset assembly, fine-tuning, and evaluation pipelines. The repository includes a merged local model under `models/lyra-merged/`, a LoRA adapter under `models/lyra-adapter/`, benchmark result JSON files under `results/`, and an assembled HuggingFace dataset under `datasets/assembled/`.

The current benchmark report is intentionally transparent: Lyra improves ARC-Challenge slightly, retains code-syntax performance, and regresses on tool-call-format versus the base model. See [BENCHMARK.md](BENCHMARK.md) for exact scores. The tool-call regression is being investigated before a broader public release.

## Model Card

### Usage

Load and run inference with the fine-tuned merged model:

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model_path = "models/lyra-merged"
device = "mps" if torch.backends.mps.is_available() else "cpu"

tokenizer = AutoTokenizer.from_pretrained(model_path)
model = AutoModelForCausalLM.from_pretrained(
    model_path,
    torch_dtype=torch.float32,
).to(device)

messages = [
    {"role": "system", "content": "You are a helpful coding assistant."},
    {"role": "user", "content": "Write a Python function to reverse a string."},
]

prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
)
inputs = tokenizer(prompt, return_tensors="pt").to(device)

with torch.no_grad():
    outputs = model.generate(**inputs, max_new_tokens=256, do_sample=False)

new_tokens = outputs[0][inputs["input_ids"].shape[1]:]
print(tokenizer.decode(new_tokens, skip_special_tokens=True))
```

### Training Parameters

Fine-tuned from `HuggingFaceTB/SmolLM2-1.7B-Instruct` using LoRA with TRL SFTTrainer. The adapter metadata and `training_args.bin` report:

| Parameter | Value |
|-----------|-------|
| LoRA rank (r) | 16 |
| LoRA alpha | 32 |
| LoRA dropout | 0.05 |
| Target modules | q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj |
| Learning rate | 2e-4 |
| Epochs | 5 in the released adapter (`scripts/train.py` default: epochs=3) |
| Batch size | 4 |
| Gradient accumulation | 4 (effective batch: 16) |
| Base model | HuggingFaceTB/SmolLM2-1.7B-Instruct |

### Limitations

- Trained on synthetic data generated by Claude Opus; behavior may not generalize beyond similar task distributions.
- Optimized for SmolLM2-1.7B chat template format; other model architectures require conversion and re-validation.
- English only; multilingual performance has not been evaluated.
- Tool calling uses SmolLM2-specific XML-wrapped JSON (`<tool_call>...</tool_call>`) rather than OpenAI function calling format.
- Current benchmark results show a tool-call-format regression versus the base model; treat the model as experimental until that regression is resolved.
